# Brainstorm Summary

- Change: add-my-orders-payment-statistics
- Date: 2026-07-20

## 确认的技术方案

- 新增独立用户页面 `/order-statistics` 和相邻侧边栏入口，不修改 `UserOrdersView.vue`、`/orders` 页面或现有 `/payment/orders/my` 契约。
- 统计页固定使用人民币 CNY，不提供币种切换、跨币种分组或换算。
- 支持最近 7/30/90 天快捷范围，以及最长 366 个自然日的自定义起止日期；自然日边界使用浏览器 IANA 时区。
- 只统计当前认证用户具有 `paid_at`、状态为 `PAID`、`RECHARGING`、`COMPLETED` 且类型为 `balance`、`usage_card`、`subscription` 的订单，金额口径为原始 `pay_amount`；不处理退款或站外手工退款修正。
- 页面展示时间范围内总实付金额、成功订单数、平均实付金额，按 `balance`、`usage_card`、`subscription` 三种类型聚合，并提供日期倒序的每日统计列表。
- 新增 `GET /api/v1/payment/orders/statistics` 汇总端点。服务层执行最长 366 天的当前用户有界字段查询，再由纯函数一次遍历生成单一人民币 `summary`、三行 `by_type` 和 `daily`。
- 新增 `GET /api/v1/payment/orders/statistics/details` 下钻端点。类型下钻传 `order_type` 和当前时间范围，日期下钻传 `date`；两种条件必须且只能选择一种。
- 类型聚合行和每日统计行整行可点击，打开只读弹窗；列表固定每页 20 条，按 `paid_at DESC, id DESC` 排序。
- 下钻 DTO 只返回订单号、订单类型、实付金额、状态、支付方式和支付时间，不提供取消、退款或其他写操作。
- 汇总与下钻复用同一用户、支付状态和半开时间边界过滤逻辑，确保聚合计数与下钻总数一致。
- 金额在聚合时转换为整数分累计，最终输出两位小数，避免浮点求和误差。
- 7/30/90 天快捷项立即查询；自定义日期使用 draft/applied 两套状态，点击查询且成功后才更新已应用范围。
- 页面首次加载失败显示重试状态；下钻失败在弹窗内重试；页面和弹窗均忽略关闭、切换或新请求之后到达的过期响应。
- 非法日期、显式非法时区、反向或超长范围、非法下钻条件返回 400，未认证返回 401。

## 关键取舍与风险

- 采用“独立汇总 API + 独立下钻 API”，避免扩展旧订单列表契约并最大限度隔离与 `main` 的冲突。
- 采用服务层有界查询和纯函数聚合，而不是 PostgreSQL 专用日期分组 SQL；代价是读取范围内的必要订单字段，通过用户过滤和 366 天上限控制内存。
- 下钻继续使用数据库过滤、计数和分页，不向前端传输全量订单。
- API 明确固定 `currency: CNY`；未来增加外币必须通过新的规格变更，而不是保留未使用的多币种 UI。
- 浏览器时区决定自然日边界，handler、service、汇总和下钻必须共享同一解析结果。
- 页面以新增文件为主，但仍需对路由、侧边栏、payment API 和 i18n 做小型追加；旧订单页面保持零修改。
- 无数据库 migration、缓存或持久化状态，回滚只需移除新入口、页面和两个端点。
